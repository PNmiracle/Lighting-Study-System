# 留学机构 CRM：第一性原理产品模型

## 系统为什么存在

CRM 不是“把 Vika 搬到网页”，而是让一次留学服务在多人、多阶段、长周期下仍然可交付、可追责、可协作。每个业务动作必须回答：服务对象是谁、当前要做什么决定、谁负责、交付物和截止时间是什么、下一步由什么事件触发。

## 核心数据边界

主数据包括 `Student`、`Staff`、`Supervisor`、`Institution`。业务对象包括：

- `ServiceCase`：一个学生购买的一次服务项目，是所有协作的总容器。
- `CaseAssignment`：某位老师在服务项目中的职责、起止时间和 SLA。
- `SupervisorRecommendation`：学生与导师之间的候选关系，存推荐理由、学生意向、反馈与排序。
- `Outreach`：某次套磁动作，存套磁信、发送时间、状态、回复和后续安排。
- `Application`：院校/项目申请，独立于套磁流程。
- `Task`：负责人、截止时间、关联对象和完成条件。
- `Deliverable`：文书、材料、签证文件等有版本的交付物。
- `Interaction`：电话、微信、邮件、会议与重要结论。

关键原则：导师是可复用的主数据；“这个学生是否选择该导师”属于 `SupervisorRecommendation`；“给该导师发了什么并收到什么回复”属于 `Outreach`。

## Vika 字段迁移

| Vika 字段 | CRM 归属 |
|---|---|
| 导师、研究领域、联系方式、主页、备注 | `Supervisor` |
| Location、学校、QS/USNEWS、Department | `Institution` / `Department` |
| 博士申请信息、其他导师信息 | `Program` 或来源链接 |
| 选导意向、你的反馈 | `SupervisorRecommendation` |
| 套磁信、确认套磁、套磁备注 | `Outreach` |
| 状态、状态变更日期、等邮件几天、后续安排、导师回复 | `Outreach` 状态机与跟进任务 |
| From、选导时间 | 创建人和审计日志 |

## 选导状态机

`需求确认 -> 导师初筛 -> 内部复核 -> 发给学生 -> 学生反馈 -> 确认套磁 -> 待发邮件 -> 等老师回复 -> Follow 未回 / 需补充材料 / 被拒绝 / 进入申请`

状态变化必须记录操作者和时间；到期未处理自动创建任务，而不是依靠老师记忆。

## 双端页面

老师端以待处理学生队列为入口，展示导师候选、事实资料、来源链接和推荐说明；支持选入方案、排序、复核、预览学生端与发送。学生端只显示精选候选，不暴露内部备注和全部导师库；学生作出 `优先套磁 / 第二批套磁 / 完全不考虑` 的明确选择并填写原因。

## 多老师协作

每个 `ServiceCase` 可以同时关联管理、沟通、选导、文书、签证老师，但每个阶段只能有一个主负责人。老师之间通过任务、交付物和结构化交接协作，不能只靠群聊消息。

## MVP 顺序

1. 学生、服务项目、老师分工与权限。
2. 导师库、推荐方案、学生反馈。
3. 套磁执行、回复状态、超时提醒。
4. 文书与材料版本管理。
5. 申请与签证流程。
6. 经营分析、顾问产能与质量指标。

Vika 在早期继续作为导师资料维护和导入来源；CRM 成为学生关系、流程状态和权限的唯一事实来源。
